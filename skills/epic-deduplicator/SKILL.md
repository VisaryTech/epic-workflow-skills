# Epic deduplicator

Очистить текст ERP-эпика от повторов и воды, не потеряв значимые требования.

## Входные данные

- `epicId | epicUrl` — идентификатор или URL эпика в ERP.
- optional: `mode`:
  - `safe-trim` — только очевидные повторы и вода;
  - `full-dedup` — полная чистка повторов и слабых формулировок;
  - `prepare-review` — чистка с акцентом на readability перед review.
- optional: `dry-run = yes|no` — по умолчанию сначала предпочитай `yes`, если пользователь явно не просил сразу сохранить.
- если во входе уже есть `epicUrl`, используй его как ссылку на эпик;
- если во входе есть только `epicId`, ссылку на эпик нормализуй только когда для этого реально доступны `ERP_BASE_URL` и `project_id`.

## ERP-конфиг

Используй локальный `python scripts/get_erp_envs.py` как канонический runtime-источник, когда для чтения, write-back или нормализации ссылки действительно нужен ERP context.

- Общие operational-правила: `../../docs/epic-skill-baseline.md`
- Lifecycle-ограничения для безопасного редактирования текста: `../../docs/epic-workflow.md`

## Коды причин

- Common reason codes бери из `../../docs/epic-reason-codes.md`.
- Для локальной диагностики допускаются короткие причины уровня `reason` / `review_notes`, например `state_label_mismatch`, `write_scope_unclear`, `semantic_drift_risk`.

## Префлайт

Перед очисткой проверь:
- есть `epicId` или `epicUrl`;
- доступно ERP read/update для эпика;
- чтение и обновление эпика будут выполняться через ERP API;
- epic существует и читается;
- доступны `../../docs`, нужные для проверки структуры;
- определён режим: `dry-run` или запись.

Если preflight не пройден, заверши run с понятным `reason` и без частичной записи.

## Процесс

1. Зафиксируй вход и работай только в режиме `erp_epic`.
2. Нормализуй вход к `epicId`.
3. Если во входе уже есть `epicUrl`, используй его как `epic_url`. Если во входе только `epicId`, а ERP-ссылка нужна и может быть корректно построена, нормализуй её через `python scripts/build_erp_url.py epic <epic_id> <project_id>`.
4. Заново прочитай актуальный epic из ERP.
5. Определи write scope: какие именно текстовые блоки подлежат очистке и какие поля трогать нельзя.
6. Разбей описание по смысловым блокам шаблона.
7. Для каждого блока выполни trim-pass:
   - найди literal duplicates;
   - найди semantic duplicates;
   - отдели signal от noise;
   - сократи vague wording;
   - сохрани distinct facts и testable statements.
8. Выполни cross-block check:
   - если один и тот же смысл дублируется в нескольких блоках, оставь его в наиболее каноничном месте;
   - в других местах либо сократи, либо оставь короткую ссылочную формулировку, если она нужна для читаемости.
9. Проверь результат против `../../docs/epic-required-fields.md` и `../../docs/epic-template.md`, чтобы не разрушить структуру.
10. Дополнительно проверь, что не потеряны обязательные поля, acceptance criteria и scope boundaries.
11. Сформируй diff summary:
   - что удалено как дубли;
   - что сокращено как вода;
   - что оставлено намеренно, хотя выглядит похожим.
12. Если `dry-run = yes`, не записывай изменения и верни предложенную редакцию/summary.
13. Если пользователь подтвердил запись или явно запросил `dry-run = no`, сохрани очищенный текст через ERP API только в разрешённом write scope без изменения lifecycle state, меток и статуса.
14. Верни краткий итог в чат.

## Формат результата

Возвращай краткий структурированный результат:
- `summary`
- `changed_blocks`
- `write_scope`
- `removed_duplicates`
- `trimmed_noise`
- `kept_intentionally`
- `review_notes`
- `write_status`

Если был `dry-run`, укажи `write_status = skipped_dry_run`.
Если запись выполнена, укажи `write_status = done`.
Если запись не удалась, укажи `write_status = failed` и краткую причину.

## Ограничения

- Не менять бизнес-смысл, scope, решение или acceptance intent.
- Не удалять уникальные факты, ограничения, зависимости, условия, edge-cases.
- Не схлопывать разные требования в одно, если теряется проверяемость.
- Удалять повтор смысла только там, где это не ломает структуру шаблона и проверяемость.
- Сокращать vague/boilerplate формулировки только без добавления новых фактов.
- При сомнении не удаляй текст, а фиксируй спорное место в `review_notes`.
- Не менять lifecycle state, статус, labels или workflow gate markers.
- Для ERP-эпика использовать только разрешённый ERP API write scope.
- Не добавлять новые требования, детали реализации или бизнес-обоснование от себя.
- Не превращать очистку в полноценный review или refiner-flow.
- Если после cleanup есть риск semantic drift, не записывать результат без явного подтверждения пользователя.
- Финальный ответ должен быть коротким.

## Что считать водой и дубликатами

Удаляй или сокращай:
- буквальные повторы одной и той же мысли;
- соседние абзацы с одинаковым смыслом;
- tautology, self-evident phrasing, boilerplate without information;
- вводные фразы без полезной нагрузки;
- чрезмерно многословные формулировки, которые можно сжать без потери смысла;
- повтор одного и того же requirement в нескольких предложениях внутри одного блока.

Не удаляй автоматически:
- acceptance criteria, даже если они похожи, пока не проверено, что они реально совпадают;
- ограничения, интеграционные условия, негативные сценарии, исключения;
- формулировки, которые различаются по actor, condition, trigger, outcome или scope.

## Lifecycle handling

- Этот skill не меняет lifecycle state и не выполняет lifecycle transition.
- Очистка текста допустима только как content-edit внутри текущего состояния эпика.
- По умолчанию запуск разрешён для active epic в любом lifecycle state, если пользователь явно просит очистку текста.
- Если lifecycle state и ERP labels расходятся, не пытайся исправлять mismatch этим skill; верни `review_notes`/`reason`, что обнаружен state-label mismatch.
- Если skill видит lifecycle-sensitive конфликт, который делает правку рискованной, предпочитай `dry-run` или stop-result вместо записи.

## Write scope

- По умолчанию редактируй только текстовые содержательные блоки эпика, где реально обнаружены дубли или вода.
- Для ERP-эпика не меняй служебные поля, lifecycle state, labels, links, relation fields, assignee, status history и другие metadata-поля.
- Не переписывай весь epic целиком, если достаточно точечного обновления отдельных блоков.
- Если API обновляет epic только whole-document replace, сначала собери точную cleaned version текущего текста без изменения остальных полей.
- Если неясно, какие поля безопасно обновлять, переходи в `dry-run` и явно фиксируй uncertainty в `review_notes`.

## Смоук-чек

- Вход: `dry-run = yes`, найдены только очевидные повторы и вода → ожидается `write_status = skipped_dry_run` с кратким diff summary.
- Вход: `dry-run = no`, write scope ясен, запись успешна → ожидается `write_status = done`.
- Вход: после trim-pass есть риск semantic drift или неясен безопасный write scope → ожидается stop-result или `skipped_dry_run` без записи.
- Вход: обнаружен state-label mismatch → ожидается `review_notes`/`reason` без попытки исправить lifecycle mismatch этим skill.
