# Epic Task Creator

## Входные данные

- `planEpicId | planEpicUrl | epicId` — идентификатор/URL дочернего plan-epic или ID родительского эпика.

Если передан только `epicId`, сначала попробуй найти дочерний plan-epic родительского epic по label `ERP_LABEL_EPIC_PLAN`.
Если plan-epic не найден, останови workflow с `❌ failed` и reason code `plan_not_found`.

## ERP-конфиг

Используй локальный `python scripts/get_erp_envs.py` как единственный канонический источник ERP-конфига.

Обязательные значения для этого skill, если текущий запуск работает с gate markers:
- `ERP_LABEL_EPIC_PLAN`
- `ERP_LABEL_APPROVED`
- `ERP_LABEL_READY`
- `ERP_BASE_URL` только если в результате нужен ERP-контекст ссылки на epic

Если обязательные значения отсутствуют, останавливай preflight с reason code этого skill.

## Модель плана

- source of truth для implementation plan — дочерний ERP epic с label `ERP_LABEL_EPIC_PLAN`;
- декомпозиция разрешена только если этот дочерний plan-epic имеет lifecycle marker `approved` через `ERP_LABEL_APPROVED`;
- `planEpicId` или `planEpicUrl` указывают именно на этот дочерний epic.

## Константы
- `TASK_LINK_TYPES`:
  - `0` = `RelatesTo`
  - `1` = `Blocks`
  - `2` = `IsBlocked`
  - Допустимы как числовые значения, так и имена.

## Источники истины

Главный source of truth для этого skill:
- implementation plan (`planEpicId | planEpicUrl`);
- текущая реализация в релевантных репозиториях.

Epic и связанные ERP-данные использовать только как:
- контейнер traceability;
- место создания и связывания dev-задач;
- источник `epicId`, если `planEpicId` выводится через поиск дочернего plan-epic.

Общие документы в `../../docs` использовать ограниченно:
- `../../docs/epic-reason-codes.md` — для стандартных reason codes;
- `../../docs/epic-workflow.md` — только если нужна минимальная проверка допустимости работы с epic/gate marker;
- `../../docs/epic-lifecycle.yaml` — для канонического lifecycle mapping и gate marker names;
- `../../docs/epic-writing-style.md` — для человеко-читаемой структуры и формулировок в ERP-комментариях и кратких summary.

Этот skill не должен опираться на канон описания эпика для содержательной декомпозиции. Содержательная декомпозиция строится по дочернему plan-epic и сверке с кодовой базой.

## Коды причин

Используй decomposition-related reason codes из `../../docs/epic-reason-codes.md`.
Локальных дополнительных reason codes этот skill не вводит.

## Префлайт

Перед стартом workflow проверь:
- есть `planEpicId`, `planEpicUrl` или `epicId`;
- дочерний plan-epic существует, доступен и не пустой;
- дочерний plan-epic имеет label `ERP_LABEL_EPIC_PLAN`;
- дочерний plan-epic имеет lifecycle marker/status `approved` через `ERP_LABEL_APPROVED`; если `approved` есть только на родительском epic, а не на child plan-epic, декомпозицию не начинать;
- обязательные ERP env/label values успешно получены через `python scripts/get_erp_envs.py`, если в текущем запуске планируется обновление gate marker на epic;
- доступен ERP API read/write для задач, связей и labels.

Если preflight не пройден — верни `❌ failed` с reason code.

## Процесс

1. Определи `planEpicId | planEpicUrl` напрямую или найди дочерний plan-epic по `epicId` и label `ERP_LABEL_EPIC_PLAN`.
2. Если в результате нужен ERP-контекст ссылки на epic, нормализуй её через `python scripts/build_erp_url.py epic <epic_id> <project_id>`.
3. Проверь, что найденный дочерний plan-epic имеет `ERP_LABEL_EPIC_PLAN` и lifecycle marker/status `approved` через `ERP_LABEL_APPROVED`.
4. Перед анализом проверь, что локальные кодовые репозитории доступны и релевантный workspace определён корректно.
5. Выполни анализ текущей реализации в релевантных локальных репозиториях с учётом локального контекста и repo-local инструкций.
6. Разбери текст дочернего plan-epic на смысловые инкременты и сверяй каждый инкремент с фактической кодовой базой.
7. Не создавай задачи, которые:
   - противоречат архитектуре или текущему устройству кода;
   - дублируют уже существующую реализацию;
   - опираются на предположения, не подтверждённые child plan-epic или кодовой базой.
8. Декомпозируй child plan-epic на несколько dev-задач с явной traceability к конкретным `section / heading / шагам`.
9. Для каждой dev-задачи зафиксируй только то, что нужно для реализации конкретного инкремента:
   - что меняем;
   - где меняем;
   - что важно по контрактам, данным, API или сущностям;
   - на какие артефакты текущей реализации опирается задача.
10. Формулируй dev-задачи кратко и по делу, без дублирования общей информации из epic/plan.
11. Не создавай задачи формата `исследовать / проанализировать / уточнить`, если implementation plan уже достаточно конкретен для выделения поставляемого инкремента.
12. Создай задачи в ERP через доступный ERP API:
   - без назначения исполнителя;
   - с обязательными `Acceptance Criteria` и `Definition of Done`.
13. Создай связи между задачами только из `TASK_LINK_TYPES` и только по смыслу зависимостей.
14. Убедись, что пройдены completion gates: tasks created, tasks verified, links created, links verified.
15. Если текущий запуск предполагает обновление gate marker на epic, после успешного прохождения completion gates установи `ERP_LABEL_READY` только через `PATCH /Epic/command/AddLabel/{epicId}`.

## Task shaping requirements

Каждая создаваемая задача должна содержать:
- traceability к конкретному `section / heading / шагу` child plan-epic;
- краткое описание поставляемого инкремента;
- `Что меняем` — 1-3 коротких пункта по сути инкремента;
- `Где меняем` — конкретные компоненты, модули, сервисы, файлы, endpoints или схемы;
- `Что по контрактам/данным` — только релевантные сущности, поля, DTO, payload, API, события или статусы;
- `Current implementation references` — конкретные ссылки на текущую реализацию;
- `Acceptance Criteria`;
- `Definition of Done`.

Правила формулировки:
- писать кратко и по делу;
- не дублировать общую информацию из epic или plan;
- фиксировать только тот контекст, который нужен для реализации конкретного инкремента;
- добавлять `In scope / Out of scope`, зависимости, ограничения и риски только если без них задача становится неоднозначной.

Избегать:
- пересказа epic или implementation plan;
- общей бизнес-мотивации, уже описанной выше;
- воды и общих фраз;
- формулировок вида `реализовать endpoint`, `добавить метод`, `написать логику`, `внести изменения в код` без описания сути инкремента.

Если child plan-epic уже достаточно конкретен, не создавать задачи формата `исследовать / проанализировать / уточнить` вместо поставляемых изменений.

Гранулярность:
- одна задача = один цельный инкремент;
- не дробить задачи слишком мелко;
- не делать чрезмерно широкие задачи из независимых инкрементов.

При создании задач:
- не назначать исполнителя;
- после создания не менять исполнителя.

## Формат результата

Формат результата:
Декомпозиция задач по плану: `<planEpicId | planEpicUrl | epicId>`
- `status:` `🔄 in_progress | ✅ done | ❌ failed`
- `epic:` нормализованный URL эпика в ERP, если epic-контекст использовался
- `lifecycle_transition:` `approved child plan-epic -> ready` (указывать, только если в рамках запуска действительно выполнялся этот переход)
- `summary:` кратко, что декомпозировано и на какие компоненты
- `tasks:` список созданных задач (`<id> <title> <url>`)
- `links:` список созданных связей (`<fromTaskId> -> <toTaskId> <type>`)
- `gate_marker:` `ERP_LABEL_READY: set | error | skipped` (обязательно указывать; для `done` может быть `skipped`, если этот запуск не предполагал изменение gate marker)

Запрещено выводить:
- внутренние рассуждения;
- длинные необработанные дампы анализа.

## Ограничения

- Не использовать описание эпика как основной источник для содержательной декомпозиции.
- Не использовать родительский epic как замену child plan-epic, если plan-epic отсутствует или пустой; в этом случае завершай run по preflight.
- Не начинать декомпозицию, если `approved` не подтверждён именно на дочернем plan-epic с label `ERP_LABEL_EPIC_PLAN`.
- Не создавать задачи без traceability к child plan-epic, ссылок на текущую реализацию, `Acceptance Criteria` и `Definition of Done`.
- Не дублировать в задаче общую информацию из epic или plan, если она не нужна для реализации конкретного инкремента.
- Формулировать задачу кратко и по делу, с фокусом на реализацию конкретного инкремента.
- По умолчанию использовать компактный шаблон: `Что меняем` -> `Где меняем` -> `Что по контрактам/данным` -> `Current implementation references` -> `Acceptance Criteria` -> `Definition of Done`.
- Не сводить задачу к шагам реализации в коде без описания сути инкремента.
- Не создавать задачи по неподтверждённым предположениям, если они не следуют из child plan-epic или текущей реализации.
- Не создавать pseudo-tasks формата `исследовать / проанализировать / уточнить`, если child plan-epic уже допускает выделение поставляемого инкремента.
- Не назначать исполнителя при создании задач и не менять исполнителя после создания.
- Не создавать связи вне допустимых типов `TASK_LINK_TYPES`.
- Не пропускать сверку с текущей реализацией.
- Не возвращать `done`, пока не пройдены ключевые completion gates.
- Если текущий запуск предполагает финальную установку `ERP_LABEL_READY`, сначала проверь допустимость такого обновления для epic.
- Для смены gate/status scoped label на epic использовать только `AddLabel`, без полного reread/merge списка labels.
- Если в текущем запуске требуется установка `ERP_LABEL_READY`, не возвращать `done`, пока marker не установлен.
- Отвечать в том же чате, где получен запрос.

## Смоук-чек

- Вход: валидный child plan-epic с label `ERP_LABEL_EPIC_PLAN` и lifecycle marker/status `approved`, локальные репозитории доступны, выполнена сверка с кодовой базой, задачи и связи успешно созданы → ожидается `✅ done`.
- Вход: родительский epic имеет `approved`, но child plan-epic с label `ERP_LABEL_EPIC_PLAN` не имеет lifecycle marker/status `approved` → ожидается `❌ failed`, reason `epic_plan_not_approved`.
- Вход: передан только `epicId`, но child plan-epic не найден → ожидается `❌ failed`, reason `plan_not_found`.
- Вход: отсутствующий или пустой child plan-epic → ожидается `❌ failed`, reason `plan_not_found`.
- Вход: child plan-epic не удалось декомпозировать без противоречий с текущей реализацией → ожидается `❌ failed`, reason из `../../docs/epic-reason-codes.md` по mismatch/validation scenario.
- Вход: agent создал задачи, несмотря на противоречие между child plan-epic и кодовой базой → ожидается `❌ failed`, reason из `../../docs/epic-reason-codes.md` по mismatch/validation scenario.
- Вход: задача не следует компактному шаблону и разрастается в длинное описание вместо краткой структуры по реализации инкремента → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задача дублирует значимые куски общего описания epic/plan вместо краткого описания конкретного инкремента → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задача без ссылок на текущую реализацию или без указания сущностей/контрактов/API там, где они существенны для инкремента → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задача сформулирована как шаги реализации (`добавить метод`, `реализовать endpoint`) без описания системного результата → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задача без AC/DoD → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задача создана с назначенным исполнителем или исполнитель изменён после создания → ожидается `❌ failed`, reason `task_validation_failed`.
- Вход: задачи/связи созданы, но в текущем запуске требовалась установка `ERP_LABEL_READY`, и marker не установился → ожидается `❌ failed`, reason `done_label_set_failed`.
