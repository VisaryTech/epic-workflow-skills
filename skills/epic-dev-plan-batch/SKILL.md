# Epic Dev Plan Batch

## Входные данные

- `epicIds[] | epicUrls[] | epics[]` — необязательный входной набор эпиков для batch-обработки.

## ERP-конфиг

Используй локальный `python scripts/get_erp_envs.py` как единственный канонический источник ERP-конфига.

Обязательные значения для этого skill:
- `ERP_LABEL_APPROVED`
- `ERP_PROJECT_ID`
- `ERP_BASE_URL`

## Источники истины

- ERP config и scope selection бери из `python scripts/get_erp_envs.py` и ERP API.
- Финальный outcome каждого элемента batch бери только из дочернего `epic-dev-plan-creator`.
- Этот skill работает как batch-orchestrator и не подменяет собой planning logic дочернего skill.

## Коды причин

- `missing_erp_config`
- `missing_epic_input`
- `epic_identification_missing`
- `epic_not_in_scope`
- `epic_deleted`
- `epic_list_fetch_failed`
- `dev_plan_batch_failed`

При `missing_erp_config` обязательно приложи список отсутствующих ключей из `python scripts/get_erp_envs.py`, например:
- `ERP_LABEL_APPROVED`
- `ERP_PROJECT_ID`
- `ERP_BASE_URL`

## Префлайт

Перед стартом проверь:
- `ERP_LABEL_APPROVED`, `ERP_PROJECT_ID` и `ERP_BASE_URL` успешно получены через `python scripts/get_erp_envs.py`;
- доступен `epic-dev-plan-creator`;
- если входной список не передан, доступно ERP-чтение списка эпиков;
- для каждого эпика можно определить `epicId` или `epicUrl`.

Если preflight не пройден, верни `❌ failed` с reason code.

## Модель выполнения

Этот skill работает как batch-orchestrator полного run по входному набору эпиков.

Правила выполнения:
- один run = один полный batch по нормализованному входному набору;
- обрабатывай эпики строго последовательно, по одному;
- не запускай следующий эпик, пока предыдущий не получил финальную классификацию;
- batch-run не считается завершённым, пока каждый элемент входного набора не классифицирован как `planned`, `needs_attention`, `failed` или `skipped`;
- итоговый batch-report формируй только после завершения полного цикла обработки входного набора;
- локальная ошибка по одному эпику не должна останавливать весь batch, если пользователь явно не попросил stop-on-first-failure.

## Процесс

1. Определи входной набор эпиков:
   - используй `epicIds[]`, `epicUrls[]` или `epics[]`, если они переданы;
   - иначе получи список эпиков с gate marker `ERP_LABEL_APPROVED` через ERP API.

2. Нормализуй вход:
   - приведи каждый элемент к виду с `epicId` и/или `epicUrl`;
   - исключи дубли;
   - неидентифицируемые эпики заноси в `failed`.

3. Проверь scope:
   - обрабатывай только эпики из выборки с gate marker `ERP_LABEL_APPROVED`;
   - эпики вне scope заноси в `skipped` с `reason code = epic_not_in_scope`.

4. Обрабатывай эпики по одному:
   - для каждого эпика отдельно вызывай `epic-dev-plan-creator`;
   - не объединяй несколько эпиков в один вызов;
   - не запускай следующий эпик, пока текущий не получил финальный outcome;
   - ошибка по одному эпику не должна останавливать batch, если пользователь не попросил обратного.

5. Зафиксируй результат по каждому эпику:
   - `planned` — `epic-dev-plan-creator` успешно подготовил implementation plan и вернул финальный success outcome;
   - `needs_attention` — `epic-dev-plan-creator` вернул финальный неуспешный результат, связанный с содержанием эпика или readiness к planning;
   - `failed` — произошла tool/runtime/process ошибка, ответ дочернего skill оборван или итог вызова не определён.

6. Сформируй единый итоговый отчет только после классификации всего входного набора.

## Формат результата

### Человеко-читаемый вывод

Используй короткий человекочитаемый итог без лишних технических деталей.

```md
# Cron // epic-dev-plan-batch

Обработано эпиков: <number>
- План готов: <number>
- Требует внимания: <number>
- Ошибка: <number>
- Пропущено: <number>

## Что готово
- <epicId> — <title>
  - короткий итог
  - plan epic: <plan_epic_link>

## Требует внимания
- <epicId> — <title>
  - что именно требует внимания

## Пропущено
- <epicId_or_ref> — <краткая причина>

## Ошибки
- <epicId_or_ref> — <краткая причина>
```

Правила:
- не выводи технические поля вроде `result:` и `total_epics:` в человекочитаемом режиме;
- не показывай пустые секции;
- детали reason code оставляй только когда они реально помогают понять проблему;
- если batch полностью успешен, секции `Пропущено` и `Ошибки` не выводи.

### Структурированный вывод

Если нужен структурированный downstream output, используй поля:
- `total_epics`
- `planned`
- `needs_attention`
- `failed`
- `skipped`
- `epics[]`
- `skipped_items[]`
- `failed_items[]`

## Ограничения

- Передавай в `epic-dev-plan-creator` только один эпик за вызов.
- Считай результат `epic-dev-plan-creator` источником истины и не переписывай содержимое плана.
- Разрешай `epic-dev-plan-creator` выполнять предусмотренные его контрактом ERP-действия после успешного прохождения completion gates.
- Не трактуй старт дочернего skill, промежуточный ответ или частичный ответ как успешное завершение.
- Если completion gates дочернего skill не подтверждены или финальный outcome неоднозначен, не классифицируй эпик как `planned`.
- Если ответ дочернего вызова оборван, неоднозначен или не позволяет надёжно определить финальный результат, классифицируй элемент как `failed` или `needs_attention` по контексту.
- Не выполняй ERP-изменения напрямую в обход `epic-dev-plan-creator`.
- Не скрывай частичные ошибки batch-обработки.
- Не останавливай весь batch из-за одной локальной ошибки, если пользователь не попросил stop-on-first-failure.
- Не возвращай итоговый batch-report до завершения полного цикла по входному списку.
- Не трактуй старт дочернего skill как завершение обработки эпика.
- Не трактуй промежуточный, частичный или оборванный ответ как `planned`.

## Смоук-чек

- Два валидных эпика с `ERP_LABEL_APPROVED`, оба успешно обработаны `epic-dev-plan-creator` → `planned=2`.
- Один валидный эпик и один эпик вне scope → один результат в `Epics`, один в `Skipped` с `epic_not_in_scope`.
- Список не передан, а получение эпиков через ERP не удалось → `❌ failed`, `reason code = epic_list_fetch_failed`.
- Один вызов `epic-dev-plan-creator` завершился ошибкой, остальные успешны → частично успешный batch-отчет без остановки всего процесса.
