# Golden epic example

Доменный эталон READY-эпика для калибровки `epic-reviewer`.

Этот пример покрывает класс задач: генерация пользовательского результата на базе уже подготовленной структурированной модели с чётко зафиксированным MVP-scope, UI-поведением, ограничениями и границами поставки.

### Бизнес-цель

**Цель:**
Обеспечить пользователю возможность автоматически формировать человекочитаемый текст требования на основании модели данных и программной логики, созданных оператором в конструкторе разметки.

**Проблема:**
Сейчас после построения модели данных и программной логики оператор вручную пишет единый текст требования в `TextView` и копирует его в карточку требования. Это увеличивает трудозатраты, замедляет работу и создаёт риск расхождений между `DMODEL`, `PLOGIC`, `TextView` в XML и содержимым `NtdRequirements.Content`.

**Ожидаемый эффект:**
- сокращено ручное составление человекочитаемого текста требования на основе модели данных и программной логики;
- снижено количество расхождений между `DMODEL`, `PLOGIC`, `TextView` и содержимым `NtdRequirements.Content`;
- повышена консистентность и воспроизводимость итогового текста нормативного требования;
- ускорена подготовка требования к дальнейшей работе оператора в карточке требования.

### AS-IS (текущее состояние)

1. Оператор открывает учетную карточку требования.
2. Нажимает кнопку для перехода к конструктору разметки в тулбаре — открывается конструктор.
3. Конструктор автоматически заполняет слои полей `MDATA` на основе данных карточки.
4. Оператор выделяет логические единицы по смыслу требования.
5. Для каждой логической единицы:
   - добавляет сущности (`Entity`), связывая их с КСИ (`KSIIdentification`);
   - добавляет сущностям свойства (`Property`), а свойствам — условия свойств (`Condition`: сравнение, равенство и др.);
   - добавляет отношения (`Relation`) для заданных сущностей и условия отношений;
   - для всех условий и отношений указывает роль: «посылка» / «следствие».
6. Система анализирует добавленные условия и отношения и автоматически генерирует для каждой логической единицы выражение для программной логики и создаёт программные правила `PRuleData` в слое `PLOGIC`.
7. Система автоматически генерирует более-менее человекочитаемую расшифровку программного правила в `PRuleText` для каждого `PRuleData`.
8. Оператор проверяет внесённые данные в модели и получившиеся программные правила, при необходимости вносит правки в модели данных.
9. Оператор на основании модели данных и программных правил для всех логических единиц самостоятельно пишет единый полный и непротиворечивый текст всего требования в блоке `TextView`, затем копирует его и вставляет в карточку требования в поле `NtdRequirement.Content`.

### TO-BE (целевое состояние)

**Общая логика:**
Данный epic заменяет ранее отменённый epic 1046 и фиксирует MVP-реализацию без расширенного async-flow, без хранения черновика результата и без автоматической повторной доставки ранее сгенерированного текста после закрытия конструктора.

Шаги подготовки модели данных и программной логики в конструкторе сохраняются без изменений. После этого пользователь на вкладке «Представления» запускает формирование человекочитаемого текста требования.

Перед запуском генерации система автоматически сохраняет актуальную версию XML разметки. После запуска генерации экран конструктора блокируется для пользователя, запустившего операцию. Пользователь не может выполнять другие действия в конструкторе, пока генерация не завершится; допускается только закрытие конструктора. При попытке закрыть конструктор система предупреждает, что результат текущей генерации текста будет потерян для пользовательского сеанса. Пока генерация выполняется, работа с этим требованием в конструкторе недоступна и для других пользователей.

После завершения генерации, если конструктор остаётся открыт, система разблокирует экран, обновляет `TextView` сформированным текстом и позволяет пользователю продолжить работу. Пользователь проверяет текст и при необходимости редактирует его вручную. Затем пользователь запускает формирование итогового XML. Система формирует XML, включая текст из `TextView`, и сохраняет этот текст в `NtdRequirements.Content`.

Если пользователь закрыл конструктор до завершения генерации, автоматически применять результат генерации при следующем открытии конструктора в рамках MVP не требуется. Если к моменту повторного открытия генерация ещё выполняется, пользователь должен видеть сообщение, что сейчас выполняется генерация текста и работа с требованием временно невозможна. Если к моменту повторного открытия генерация уже завершилась, пользователь может продолжить работу с требованием, при этом автоматического обновления текста ранее завершённым результатом не происходит.

**Правила / ограничения:**
- генерация человекочитаемого текста должна основываться на данных слоёв `DMODEL` и `PLOGIC`;
- перед запуском генерации система должна автоматически сохранить актуальную версию XML разметки;
- на время генерации экран конструктора блокируется для пользователя, запустившего операцию;
- на время генерации требование недоступно для работы в конструкторе другим пользователям;
- при попытке закрыть конструктор во время генерации система должна показать предупреждение о потере результата текущей генерации для пользовательского сеанса;
- в рамках MVP не требуется хранение и повторная доставка ранее сгенерированного текста после закрытия конструктора;
- формируемый текст должен быть на русском языке, в официально-деловом стиле, без явных грамматических и орфографических ошибок, полным, точным и непротиворечивым относительно данных `DMODEL` и `PLOGIC`.

### UI / Отображение

**Зона изменений:**
существующий конструктор разметки, левая панель «Метаданные и представления», вкладка «Представления», область текстового представления `TextView`.

**Структура / порядок блоков:**
- на вкладке «Представления» рядом с текстом текстового представления размещается кнопка формирования человекочитаемого текста всего требования;
- после запуска генерации конструктор переходит в заблокированное состояние;
- при попытке закрыть конструктор во время генерации система показывает предупреждение;
- после завершения генерации, если конструктор открыт, текст отображается в `TextView`, а конструктор разблокируется.

**Требования к отображению:**
- кнопка формирования текста должна быть доступна в зоне работы с `TextView`;
- во время генерации элементы редактирования конструктора недоступны пользователю, запустившему операцию;
- если другой пользователь открывает конструктор этого же требования во время генерации, он видит сообщение, что выполняется генерация текста и работа с требованием временно невозможна;
- если пользователь, запустивший генерацию, повторно открывает конструктор до завершения операции, он также видит сообщение, что генерация ещё выполняется и работа с требованием временно невозможна;
- если `TextView` уже заполнен, перед запуском генерации система показывает предупреждение о замене текста и предлагает продолжить или отменить действие.

### Пользовательские сценарии

**Основной сценарий:**
1. Оператор открывает карточку требования и переходит в конструктор разметки.
2. Оператор выполняет разметку требования, формирует сущности, свойства, отношения, условия и получает программные правила.
3. Оператор переходит на вкладку «Представления» и нажимает кнопку формирования человекочитаемого текста всего требования.
4. Если в `TextView` уже есть текст, система предупреждает о его замене; оператор подтверждает продолжение или отменяет действие.
5. Система автоматически сохраняет актуальную версию XML разметки.
6. Система запускает генерацию текста на основе `DMODEL` и `PLOGIC`.
7. Система блокирует экран конструктора для пользователя до завершения генерации.
8. После завершения генерации система разблокирует конструктор и обновляет `TextView` сформированным текстом.
9. Оператор проверяет текст, при необходимости вручную редактирует его.
10. Оператор запускает формирование итогового XML.
11. Система формирует XML, включая текст из `TextView`, и сохраняет текст в `NtdRequirements.Content`.

**Альтернативные сценарии:**
- пользователь отменяет замену существующего текста в `TextView` — формирование нового текста не выполняется;
- во время генерации второй пользователь открывает конструктор того же требования и получает сообщение, что выполняется генерация текста, поэтому работа с требованием временно невозможна;
- пользователь, запустивший генерацию, закрывает конструктор до завершения операции; перед закрытием система предупреждает, что результат текущей генерации текста будет потерян для пользовательского сеанса;
- пользователь, запустивший генерацию, повторно открывает конструктор до завершения операции и получает сообщение, что генерация ещё выполняется, поэтому работа с требованием временно невозможна;
- пользователь закрыл конструктор, а генерация завершилась в его отсутствие; при повторном открытии конструктора автоматического применения ранее сгенерированного текста не происходит, пользователь может продолжить работу с требованием в обычном режиме.

### Acceptance Criteria

- в конструкторе разметки на вкладке «Представления» в левой панели «Метаданные и представления» рядом с текстом текстового представления доступна кнопка формирования человекочитаемого текста всего требования;
- если `TextView` уже заполнен, перед запуском генерации система показывает предупреждение о замене текста и предлагает выбрать: продолжить или отменить действие;
- при запуске генерации система автоматически сохраняет актуальную версию XML разметки;
- после запуска генерации экран конструктора блокируется, и пользователь не может продолжать редактирование до завершения операции;
- во время генерации другой пользователь не может работать с этим требованием в конструкторе и получает сообщение о временной недоступности;
- если пользователь, запустивший генерацию, повторно открывает конструктор до завершения операции, он получает сообщение, что генерация ещё выполняется и работа с требованием временно невозможна;
- если пользователь пытается закрыть конструктор во время генерации, система показывает предупреждение о потере результата текущей генерации для пользовательского сеанса;
- после завершения генерации, если конструктор открыт, система разблокирует экран и обновляет `TextView` сформированным текстом;
- сформированный текст создаётся на основе данных `DMODEL` и `PLOGIC`, отображается на русском языке в официально-деловом стиле и не противоречит данным модели и логики;
- после успешного формирования итогового XML текст из `TextView` включается в XML и записывается в `NtdRequirements.Content`;
- если пользователь закрыл конструктор до завершения генерации, при повторном открытии автоматическая подстановка ранее сгенерированного текста в рамках MVP не выполняется.

### Scope (входит в задачу)

- добавить кнопку в существующий конструктор разметки в левую панель «Метаданные и представления» на вкладку «Представления» рядом с текстом текстового представления;
- реализовать интеграцию через API с сервисом генерации человекочитаемого текста на основе данных конструктора;
- поддержать MVP-сценарий блокирующего запуска генерации: пользователь запускает генерацию и ждёт результат при заблокированном экране конструктора;
- автоматически сохранять актуальную версию XML перед запуском генерации;
- блокировать работу с требованием в конструкторе для других пользователей на время генерации;
- после получения результата обновлять `TextView`; при формировании XML включать этот текст в итоговый файл и сохранять в `NtdRequirements.Content`.

### Out of Scope (не входит)

- реализация самой нейросетевой модели перевода данных конструктора в человекочитаемый формат;
- расширенный async-flow со статусами «в очереди» / «выполняется» / «готово»;
- хранение промежуточного или отложенного результата генерации как черновика;
- автоматическая повторная доставка или автоподстановка ранее сгенерированного текста после повторного открытия конструктора;
- обработка устаревания результата относительно новой версии XML после закрытия конструктора.

### Ограничения

- в рамках данного epic не входит реализация самой нейросетевой модели перевода данных в человекочитаемый формат; предполагается использование внешнего/смежного сервиса через API;
- генерация текста должна основываться только на данных, уже сформированных в слоях `DMODEL` и `PLOGIC`, без ручного конструирования текста системой вне этой модели;
- решение должно быть встроено в существующий конструктор разметки и не должно ломать текущий сценарий подготовки модели и программной логики;
- для MVP реализуется только блокирующий сценарий генерации в рамках одного пользовательского запуска без расширенного жизненного цикла фоновой обработки результата.

### Риски и допущения

- предполагается, что качества данных в `DMODEL` и `PLOGIC` достаточно для автоматической генерации полного и непротиворечивого текста требования;
- есть риск, что результат генерации потребует ручной пост-редакции оператором из-за стилистических, терминологических или логических неточностей;
- есть риск задержек или ошибок при обращении к внешнему сервису генерации текста, из-за чего пользователь может дольше ожидать разблокировки конструктора;
- предполагается, что ручное редактирование текста после генерации является допустимым сценарием и не должно автоматически менять данные модели/логики без повторного формирования XML;
- есть риск, что при закрытии конструктора во время генерации пользователь потеряет результат текущего запуска и будет вынужден инициировать генерацию повторно.

### Зависимости

- доступность API сервиса/компонента, выполняющего преобразование данных конструктора в человекочитаемый текст;
- наличие корректно сформированных и доступных для обработки данных в слоях `DMODEL`, `PLOGIC`, `VIEW`;
- механизм автоматического сохранения XML перед запуском генерации;
- механизм формирования итогового XML и запись текста в `NtdRequirements.Content`;
- UI-расширение существующего конструктора разметки в части вкладки «Представления»;
- механизм блокировки работы с требованием в конструкторе на время выполнения генерации.
