# Epic Workflow

## Обзор

Используй этот скилл как основную точку входа для работы с эпиками в `epic-workflow`.
Он направляет запрос в подходящий локальный дочерний скилл, использует внутренние `docs/` как общую доменную базу и не даёт случайно откатываться к путям из исходного донорского репозитория.

## Общий Контекст

- Рассматривай локальную папку `docs/` как общую доменную базу для правил epic workflow, lifecycle gates, обязательных полей, reason codes, стиля формулировок и шаблонов.
- Рассматривай `docs/epic-glossary.md` как словарь терминов и preferred wording для `docs/`, `SKILL.md` и reference-файлов.
- Рассматривай `docs/epic-lifecycle.yaml` как машинно-читаемый источник истины для status order, transitions, summary groups и mapping `status -> ERP_LABEL_*`.
- Считай status-related ERP env keys производными от `docs/epic-lifecycle.yaml`; не дублируй их статическими списками в skill-контрактах или вспомогательных скриптах.
- Рассматривай `docs/epic-workflow.md` как человекочитаемый источник истины для process rules и смысла lifecycle stages.
- Рассматривай дочерние скиллы в `skills/` как исполняемый capability-слой этого пакета.
- Предпочитай относительные пути внутри пакета: `skills/...` и `docs/...`.
- Разрешай внешние зависимости вроде `tasktracker-api` через установленные Codex skills, а ERP-конфиг читай из локального `scripts/get_erp_envs.py` внутри этого репозитория.

## Маршрутизация

Выбирай дочерний скилл по намерению пользователя:

- Новый intake эпика и создание в ERP:
  используй `skills/epic-creator`.
- Очистка текста эпика от дублирования и многословия без изменения бизнес-intent:
  используй `skills/epic-deduplicator`.
- Доработка эпика после замечаний, особенно в потоке `to-define`:
  используй `skills/epic-refiner`.
- Review готовности эпика и решение `approved | to-define`:
  используй `skills/epic-reviewer`.
- Подготовка implementation plan:
  используй `skills/epic-dev-plan-creator`.
- Декомпозиция implementation plan в ERP dev-задачи:
  используй `skills/epic-task-creator`.
- Batch-планирование или batch-декомпозиция для нескольких эпиков:
  используй `skills/epic-dev-plan-batch` или `skills/epic-dev-task-batch`.
- Статусный отчёт по workflow эпиков:
  используй `skills/epic-status-report`.
- Оценка веса всех задач внутри эпика:
  используй `skills/epic-task-weight-estimator`.

Если пользователь в общем виде просит "поработать с эпиком", но точное намерение не указано, разбирай запрос в таком порядке:
1. Создать новый эпик.
2. Улучшить или очистить существующий текст эпика.
3. Проверить готовность эпика.
4. Подготовить или обновить implementation plan.
5. Создать dev-задачи по плану.
6. Сформировать статусный или аналитический отчёт.

## Базовый Workflow

Типовой порядок жизненного цикла внутри этого пакета:
1. Создать epic через `epic-creator`.
2. При необходимости очистить текст через `epic-deduplicator`.
3. Выполнить review через `epic-reviewer`.
4. Доработать через `epic-refiner`, если эпик вернулся в `to-define`.
5. Подготовить план через `epic-dev-plan-creator`.
6. Декомпозировать план через `epic-task-creator`.
7. Запустить `epic-task-weight-estimator`, если далее нужны оценки task weight.

Используй batch-скиллы только тогда, когда пользователь явно просит обработать много эпиков за один запуск.

## Ограничения

- Не обходи дочерние скиллы, если нужная операция уже покрывается локальным специализированным скиллом.
- Не дублируй общие epic-правила в каждом ответе; загружай только те документы, которые действительно нужны.
- Не рассматривай этот umbrella-скилл как замену внешним зависимостям вроде `tasktracker-api`; если выбранный дочерний скилл требует ERP-доступ, считай обязательными локальный ERP-конфиг из `config/erp-env.json` и внешние интеграции ERP API.
