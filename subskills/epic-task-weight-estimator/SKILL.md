# Epic Task Weight Estimator

## Входные данные

- `epicId | epicUrl` — идентификатор или ссылка на эпик в ERP.
- `force_reestimate` — необязательный флаг принудительного пересчёта даже если у задачи уже есть `weight`.

## ERP-конфиг

- `python scripts/get_erp_envs.py` — обязательный локальный источник ERP-конфига.
- `../../docs/epic-lifecycle.yaml` — канонический lifecycle mapping для `ready` lifecycle state и plan-epic marker.

## Источники истины

- текст эпика — верхняя граница scope и основной источник истины;
- child plan-epic, если найден, уточняет ожидаемую реализацию;
- текст задачи задаёт локальный scope;
- при конфликте между epic и task scope приоритет у текущего текста эпика.

## Константы

- `READY_STATUS` = `ready`

## Коды причин

- `missing_epic_input`
- `epic_not_found`
- `epic_not_ready`
- `epic_tasks_not_found`
- `task_text_missing`
- `task_weight_field_not_available`
- `task_weight_update_failed`
- `task_comment_failed`
- `erp_config_missing`

## Префлайт

Перед стартом workflow проверь:
- есть `epicId` или `epicUrl`;
- ERP config успешно получен через `python scripts/get_erp_envs.py`;
- доступен ERP API read/write для задач и комментариев;
- эпик существует и доступен для чтения;
- эпик имеет expected gate marker `ERP_LABEL_READY` и соответствует lifecycle state `READY_STATUS`;
- список задач эпика доступен для чтения;
- скрытые (`hidden`) задачи можно отфильтровать и исключить из оценки;
- у задач доступно поле `weight`;
- доступны права на обновление веса задач и добавление комментариев.

Отсутствие дочернего plan-epic или label `ERP_LABEL_EPIC_PLAN` не блокирует выполнение.

Если preflight не пройден, верни `❌ failed` с одним reason code.

## Правила оценки

### Базовая формула

- `base_hours = research_hours + implementation_hours`
- `final_weight = base_hours + risk_adjustment_hours`
- в ERP поле `weight` сохраняется `final_weight`

### Разбиение часов

В `research_hours` включать чтение и сопоставление требований, разбор текущей логики, поиск точки изменения в коде, уточнение интеграций и зависимостей, подготовку решения.

В `implementation_hours` включать изменение кода и конфигурации, миграции, контрактные и API-изменения, тестирование, исправление ожидаемых интеграционных дефектов, доведение результата до готового состояния в рамках scope задачи.

### Приоритет scope

- если задача уже, чем эпик, оценивать по задаче;
- если задача конфликтует с эпиком, приоритет у эпика.

### Единицы и округление

- использовать только целые часы;
- минимальный ненулевой вес — `1`;
- округлять итог вверх до целого часа.

### Практическая шкала

- `1-2` ч — простое локальное изменение без заметных неизвестных;
- `3-5` ч — обычная задача с понятной точкой входа;
- `6-12` ч — несколько связанных изменений, интеграция или нетривиальная логика;
- `13-24` ч — сложная задача с несколькими зонами системы;
- `24+` ч — кандидат на дополнительную декомпозицию.

### Оценка риска

Сначала оцени `base_hours`, затем добавь `risk_adjustment_hours` только при явной неопределённости.

Факторы риска: неизвестная часть кода, внешняя интеграция, слабое или противоречивое описание, зависимость от другой команды, вероятные side effects, требования обратной совместимости.

Уровни риска:
- `low` — понятный scope, известная зона кода, нет внешних зависимостей;
- `medium` — есть 1-2 неизвестных, интеграция или неполный контекст;
- `high` — новая зона системы, много неизвестных, сложная интеграция или критичный контур.

Корректировка риска:
- `low` → `+0%` к `base_hours`;
- `medium` → `+15-25%` к `base_hours`;
- `high` → `+30-50%` к `base_hours`.

Если `final_weight > 24`, считай задачу кандидатом на декомпозицию.

## Процесс

1. Нормализуй `epicId | epicUrl` и получи ERP config через `python scripts/get_erp_envs.py`.
2. Прочитай актуальный текст эпика из ERP и используй его как основной источник business scope.
3. Проверь, что эпик имеет gate marker `ERP_LABEL_READY` и соответствует lifecycle state `READY_STATUS`. Если нет, остановись с `reason code = epic_not_ready`.
4. Попробуй найти и прочитать дочерний plan-epic по label `ERP_LABEL_EPIC_PLAN`.
5. Если child plan-epic найден, используй его как дополнительный контекст реализации.
6. Если child plan-epic не найден, продолжай без fail по связке текста эпика и текста задачи.
7. Получи все связанные с эпиком задачи из ERP.
8. Исключи из оценки скрытые (`hidden`) задачи и другие явные технические/system задачи.
9. Если `force_reestimate` не задан и у задачи уже есть `weight`, пропусти её и зафиксируй пропуск в итоге.
10. Для каждой оставшейся задачи прочитай текст и проверь, что содержания достаточно для оценки. Если нет, фиксируй `reason code = task_text_missing`.
11. Для каждой валидной задачи рассчитай `research_hours`, `implementation_hours`, `base_hours`, `risk_level`, `risk_adjustment_hours`, `final_weight`.
12. Сохрани `final_weight` в поле `weight` через ERP endpoint `PATCH /Task/command/ChangeWeight/{taskId}`.
    - Используй `visary-cloud-api-skills` и его TaskTracker API capability; не запускай напрямую внешние API CLI-скрипты.
13. После успешного обновления веса обязательно добавь комментарий с деталями расчёта через ERP endpoint `POST /TaskComment/command/Create`.
14. Используй единый шаблон комментария из `references/weight-comment-template.md`.
15. Если комментарий добавить не удалось, не считай задачу успешно обработанной и фиксируй `reason code = task_comment_failed`.
16. После обработки всех задач посчитай количество обновлённых и пропущенных задач и общий вес эпика.
17. Верни короткий итоговый результат в чат.

## Шаблон комментария

Используй единый шаблон комментария в задаче:

```md
Оценка веса задачи

- research_hours: <value>
- implementation_hours: <value>
- base_hours: <value>
- risk_level: <low | medium | high>
- risk_adjustment_hours: <value>
- final_weight: <value>

Факторы оценки:
- <factor 1 | none>
- <factor 2>

Допущения:
- <assumption 1 | none>
- <assumption 2>
```

Если факторов или допущений нет, указывай `- none`.

## Формат результата

### Успех

Вернуть только:

```md
✅ Вес задач эпика обновлён: <epic_name>
Эпик: <epic_link>
Обновлено задач: <updated_count>
Вес эпика, ч: <total_hours>
```

### Частичный успех

Если часть задач не удалось оценить или обновить, вернуть только:

```md
⚠️ Вес задач эпика обновлён частично: <epic_name>
Эпик: <epic_link>
Обновлено задач: <updated_count>
Пропущено задач: <skipped_count>
Вес эпика, ч: <total_hours>
reason: <reason>
```

### Ошибка

Вернуть только:

```md
❌ Вес задач эпика не обновлён: <epic_name>
Эпик: <epic_link>
reason: <reason>
```

Если URL эпика определить не удалось, строку `Эпик:` можно опустить.

## Ограничения

- Не оценивать эпики вне gate marker `ERP_LABEL_READY` и вне lifecycle state `READY_STATUS`.
- Не считать отсутствие child plan-epic критической ошибкой.
- Не выносить core-логику расчёта в helper script, локальный калькулятор или отдельный внешний раннер.
- Не оценивать скрытые (`hidden`) и явные технические/system задачи.
- Не менять у задач статусы, labels, assignee, sprint, milestone и другие поля кроме `weight` и комментария с деталями расчёта.
- Не создавать новые задачи автоматически.
- Не перезаписывать уже заполненный `weight`, если явно не задан `force_reestimate`.
- Не подменять текст задачи догадками, если содержания недостаточно для оценки.
- Обязательный completion gate для каждой задачи: `weight updated` + `comment added`.
- Не считать обновление задачи завершённым, если вес записан, но комментарий с деталями расчёта не добавлен.
- Не записывать в `weight` дробные или текстовые значения, только целое число часов.
- Не добавлять риск как произвольную «подушку» без явного фактора неопределённости.
- При конфликте между epic и task scope приоритет отдавать текущему тексту эпика как источнику истины.
- Если итоговый `final_weight > 24`, помечать задачу как кандидата на декомпозицию только в комментарии, без автоматического изменения самой задачи.

## Смоук-чек

- Вход: эпик имеет `ERP_LABEL_READY`, lifecycle state `ready`, plan найден, задачи имеют достаточный текст, update веса успешен → ожидается `✅ Вес задач эпика обновлён`.
- Вход: эпик без `ERP_LABEL_READY` или в lifecycle state отличном от `ready` → ожидается `❌ ... reason: epic_not_ready`.
- Вход: plan отсутствует, но эпик и задачи доступны → ожидается success или partial success без fail по отсутствию плана.
- Вход: часть задач hidden/system → ожидается success или partial success без их оценки.
- Вход: часть задач уже имеет `weight`, `force_reestimate` не задан → ожидается partial success с пропуском таких задач.
- Вход: задача без достаточного текста → ожидается partial success или fail с `reason: task_text_missing`.
- Вход: endpoint обновления веса вернул ошибку → ожидается `❌` или `⚠️` с `reason: task_weight_update_failed`.
- Вход: вес обновился, но комментарий в задачу не добавился → ожидается `❌` или `⚠️` с `reason: task_comment_failed`.
