# Epic Naming

Единые правила именования эпиков.

## Базовый шаблон

`Подсистема / Блок_или_Модуль / Функционал`

## Цель названия

Название должно:
- быстро объяснять, о чём эпик
- показывать функциональную область
- быть достаточно коротким для чтения в списках
- отличать эпик от соседних инициатив

## Правила

1. Использовать структуру из трёх частей:
   - подсистема
   - блок или модуль
   - краткое описание функционала
2. Описание функционала делать коротким и предметным.
3. По возможности ограничивать функциональную часть 3-4 словами.
4. Не использовать размытые формулировки без предметного смысла.
5. Не использовать слова вроде:
   - `доработка`
   - `изменения`
   - `улучшение`
   - `оптимизация`
   если из названия не ясно, что именно меняется.
6. Не перегружать название деталями реализации.
7. Не включать в название длинные пояснения, риски, условия и служебные комментарии.

## Good examples

- `Закупки / Заявки / Согласование по маршруту`
- `Склад / Инвентаризация / Массовая корректировка остатков`
- `Продажи / Договоры / Контроль срока действия`
- `Финансы / Платежи / Импорт банковской выписки`

## Bad examples

- `Доработка заявок`
- `Изменения в складе`
- `Оптимизация процесса`
- `Новый функционал по договорам`

## Notes for agents

1. Если пользователь дал слишком общее название, агент должен предложить более предметный вариант.
2. Если модуль не определён, агент должен сначала уточнить контекст, а не придумывать название.
3. Название должно отражать понятную функциональную цель, а не внутренний процесс обсуждения.
