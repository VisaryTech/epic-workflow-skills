# Epic Naming

Единые правила именования business epic.

## Template

Title всегда оформляется так:

`Подсистема / Блок_или_Модуль / Функционал`

Пример:

`РТ / Карточка требования / Контроль срока действия`

## Rules

Title должен:
- состоять ровно из 3 непустых частей, разделённых `/`;
- использовать понятную подсистему;
- использовать понятный блок или модуль;
- содержать короткий предметный функционал;
- быть достаточно коротким для чтения в списках;
- отличать epic от соседних инициатив.

Функциональную часть по возможности ограничивать 3-4 словами.

## Forbidden

В title нельзя включать:
- длинные пояснения;
- бизнес-условия;
- сценарные детали;
- критерии приёмки;
- детали реализации;
- риски и служебные комментарии;
- фразы вида `если`, `при`, `по дате`, `о необходимости`.

Эти сведения должны быть в описании epic, а не в title.

## Bad title signals

Title считается плохим, если:
- не соответствует формату с `/`;
- содержит меньше или больше 3 частей;
- похож на полное предложение;
- описывает обсуждение, а не функционал;
- использует размытые слова без конкретизации: `доработка`, `изменения`, `улучшение`, `оптимизация`.

## ERP save gate

Перед create-stage агент обязан явно зафиксировать:
- `Название epic`;
- `Краткое описание epic`.

Если title не прошёл naming-check, создание epic в ERP не выполняется.

Reason code: `naming_not_stable`.

Operational rule: сначала структура названия, потом сохранение в ERP.

## Examples

Good:
- `Закупки / Заявки / Согласование по маршруту`
- `Склад / Инвентаризация / Массовая корректировка остатков`
- `Продажи / Договоры / Контроль срока действия`
- `Финансы / Платежи / Импорт банковской выписки`

Bad:
- `Доработка заявок`
- `Изменения в складе`
- `Оптимизация процесса`
- `Новый функционал по договорам`
- `РТ / Карточка требования / Контроль срока действия по дате окончания если требование активно`
- `РТ / О необходимости контроля срока действия`
