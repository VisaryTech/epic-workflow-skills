# Epic writing style

Этот документ задаёт единые правила человеко-читаемого оформления ERP-эпиков и комментариев к ним.

Термины и preferred wording брать из `docs/epic-glossary.md`.

Использовать его как источник истины по стилю для skills, которые:
- создают epic;
- редактируют epic;
- публикуют review/comment/result в ERP;
- формируют implementation-plan related комментарии.

## Core principles

- Писать коротко и по делу.
- Предпочитать списки сплошному тексту.
- Не повторять один и тот же смысл в нескольких разделах.
- Не подменять требования рассуждениями или лишними реализационными деталями.
- Писать так, чтобы читатель быстро понял: проблему, целевое изменение, границы `scope` и способ проверки результата.

## Epic description style

Сохраняй ясность и различимость смысловых зон epic description:
- контекст / проблема;
- цель;
- текущее и целевое состояние;
- основной сценарий;
- acceptance criteria;
- границы `scope`;
- зависимости / ограничения;
- риски / open questions.

### Section rules

#### Контекст / проблема

Писать:
- что сейчас не так;
- кого это затрагивает;
- почему изменение нужно бизнесу или процессу.

Не писать:
- длинную предысторию без влияния на решение;
- технические детали, если они не важны для понимания проблемы.

#### Цель

Формулировать как целевое изменение состояния.

Хорошо:
- "Система должна позволять ..."
- "Нужно обеспечить ..."

Плохо:
- "Хотим как-нибудь улучшить ..."
- "Надо подумать над возможностью ..."

#### As is / To be

Разделять текущее и целевое состояние явно.

Требования:
- `As is` описывает только текущее поведение или ограничения.
- `To be` описывает только ожидаемое поведение после изменения.
- Не смешивать текущую и целевую логику в одном абзаце.

#### Основной сценарий

Описывать основной пользовательский или бизнес-сценарий пошагово, если это снимает неоднозначность:
- обычно 3-7 шагов;
- шаги должны быть наблюдаемыми;
- не подменять сценарий внутренней реализацией.

#### Acceptance criteria

Acceptance criteria должны быть проверяемыми, конкретными и наблюдаемыми.

Хороший формат:
- "Если ..., то ..."
- "При ..., система ..."
- "Пользователь может ..."

Плохой формат:
- "Сделать удобно"
- "Реализовать поддержку"
- "Система должна работать корректно"

#### Scope boundaries

Границы scope должны быть зафиксированы явно.

Если workflow или шаблон предполагает разделение `In scope` и `Out of scope`, использовать его.
Если нет, всё равно нужно явно показать, что входит в изменение, а что не входит.

#### Зависимости / ограничения

Фиксировать только реальные зависимости и ограничения:
- внешние системы;
- связанные команды;
- обязательные интеграции;
- ограничения по данным, процессам, ролям, срокам.

#### Риски / open questions

Не маскировать неизвестность под уверенное требование.

Если данных не хватает:
- выносить вопрос явно;
- не придумывать факт вместо уточнения.

## ERP comment style

Комментарии в ERP должны быть короткими и ориентированными на действие.

### Comment principles

- Сначала смысл, потом служебные детали.
- Первые строки должны давать понятный итог.
- Служебные поля допустимы только если они реально помогают workflow.
- Не превращать комментарий в сырой system dump.

## Recommended comment formats

### Review comment

Рекомендуемая структура:

1. Вердикт
2. Критичные замечания
3. Некритичные замечания
4. Что нужно для следующего шага

Если замечаний нет:
- явно писать, что критичных замечаний не найдено.

### Refine / update comment

Рекомендуемая структура:

1. Что обновлено
2. Что осталось открытым
3. Готовность к следующему шагу

### Plan comment

Рекомендуемая структура:

1. Итог
2. Что покрывает план
3. Ссылка на план
4. Что требуется дальше

### Task decomposition comment

Рекомендуемая структура:

1. Итог
2. Что декомпозировано
3. Сколько задач создано
4. Следующий шаг workflow

## Human-readable wording rules

- Предпочитать простые глаголы: `обеспечить`, `разделить`, `показать`, `проверить`, `создать`, `обновить`.
- Избегать канцелярита: `в рамках`, `с целью`, `необходимо произвести`, `осуществить доработку`.
- Избегать расплывчатых слов: `корректно`, `удобно`, `гибко`, `качественно` без конкретики.
- Если мысль можно выразить списком, использовать список.

## Разделение человекочитаемого и машинно-читаемого контента

Если workflow требует служебные поля:
- в начале комментария давать человекочитаемое summary;
- ниже оставлять только минимально нужный служебный блок.

Не ставить в начало комментария `run status`, `reason code`, `sync status`, если без них итог уже понятен.

## Skill integration guidance

Skills, которые создают или меняют epic/comment, должны:
- ссылаться на этот документ как на источник истины по стилю;
- использовать его для читаемости, плотности текста и выбора формулировок;
- не дублировать весь style guide внутри `SKILL.md`, кроме коротких role-specific reminders.
